## 1. 背景

`duet` 源自 `fuwari`，但当前目标已经不再是“把博客模板继续做大”，而是把它演进为：

- 程序与内容分离
- 公开与私密内容分离
- 以内容域而不是单一博客心智组织页面
- 兼顾公开静态产物与作者态受保护内容

与此同时，`Firefly` 作为基于 `Fuwari` 的重度二开项目，已经在主题产品化、配置颗粒度、页面模块和阅读体验上走得很远。

因此本设计的目的不是“复刻 Firefly”，而是回答一个更具体的问题：

> 哪些能力可以被 `duet` 吸收，以增强产品完成度，而又不破坏 `duet` 既定的 private-first 架构方向？

## 2. 核心结论

`Firefly` 值得借鉴的，不是“功能越多越好”的主题策略，而是以下三类能力：

1. 配置与页面系统的产品化
2. 阅读与展示体验的增强
3. 将常见站点模块沉淀为可复用能力

但 `duet` 必须明确拒绝两类方向：

1. 回到“博客主题中心型”代码结构
2. 为了视觉热闹而牺牲公开 / 私密边界与架构克制

一句话概括：

- 从 `Fuwari` 保留克制
- 从 `Firefly` 吸收产品化能力
- 由 `duet` 自己坚持 private-first 与内容引擎边界

## 3. 三者定位差异

| 项目 | 本质定位 | 主要目标 |
| --- | --- | --- |
| `Fuwari` | 轻量静态博客模板 | 快速搭建公开博客 |
| `Firefly` | 重度增强博客主题 | 提供更强的配置能力与页面功能 |
| `duet` | private-first 个人门户引擎 | 支持双仓库、私密内容与多内容域 |

因此，`Firefly` 对 `duet` 的参考价值主要在“主题工程化经验”，而不在“博客主题全量功能”。

## 4. 吸收原则

`duet` 在吸收 `Firefly` 能力时，必须同时满足以下原则：

## 4.1 不破坏内容域边界

能力的落点应优先进入以下层次之一：

- `site` 域配置
- `blog` 域组件
- `moments` 域组件
- 公共布局与展示原语

不能因为借鉴某个主题功能，就重新把全站逻辑收缩回“所有东西都是博客附属物”。

## 4.2 不破坏公开 / 私密边界

任何能力若会影响以下链路，都必须先经过 `duet` 的公开边界审查：

- RSS
- sitemap
- Pagefind
- 静态构建产物
- 公开结构化数据

原则上，凡是可能让私密内容出现在静态产物中的实现方式，都不能直接照搬 `Firefly`。

## 4.3 不把系统契约误做成主题配置

以下内容属于系统契约，不应主题化：

- 内容来源模式
- 内容 schema
- 私密内容可见性
- 鉴权模型
- public manifest / protected manifest 边界

`Firefly` 式的大量配置开关，适合用于视觉与展示，不适合覆盖系统边界。

## 4.4 优先吸收“原语”，而不是“页面成品”

`duet` 更应该吸收：

- 布局模式
- 卡片样式
- widget 抽象
- 配置组织方式

而不是直接吸收：

- 某个具体友链页 UI
- 某个具体音乐播放器页
- 某个具体相册页模板

原语可以服务未来的 `home / moments / blog / portfolio`，页面成品则容易把系统拖回主题堆料路线。

## 5. 可直接吸收的能力

这些能力与 `duet` 当前方向高度兼容，适合进入近期路线。

## 5.1 配置拆分

当前 `duet` 仍把大量配置集中在单个 `src/config.ts` 中。随着 `home / moments / private posts / cv / portfolio` 增长，这个文件会很快膨胀。

建议吸收 `Firefly` 的做法，将配置拆分为多个稳定模块，例如：

- `config/site.ts`
- `config/profile.ts`
- `config/navigation.ts`
- `config/theme.ts`
- `config/license.ts`
- `config/home.ts`
- `config/widgets.ts`

这是最值得优先吸收的能力，因为它强化的是“公开引擎配置接口”，不是主题噱头。

## 5.2 布局原语

`Firefly` 的价值不在于某一种具体博客布局，而在于它已经证明：

- 列表
- 网格
- 双侧栏
- 单侧栏
- 时间流

可以被设计成稳定的布局模式。

`duet` 应吸收的不是“把首页做成 Firefly 风格”，而是把布局抽象成可复用原语，服务：

- 博客首页
- 归档页
- Moments 时间流
- 未来 Home 门户区块

## 5.3 阅读体验增强

这部分与 `duet` 的博客目标天然一致，可以直接吸收：

- 更成熟的 callout / admonition 表达
- 更细的代码块视觉增强
- 更稳定的仓库卡片 / 外链卡片模式
- 更完整的封面图展示策略
- 更细的 Markdown 文章展示细节

这类能力增强的是“长文阅读质量”，不会破坏 private-first 架构。

## 5.4 主题与字体能力

适合吸收的内容包括：

- 字体配置拆分
- 主题色与背景模式的清晰配置项
- 可控的展示风格切换

这些能力强化的是“站点人格”，并且对未来的 `home` 设计也有帮助。

## 6. 需要改造后再吸收的能力

这些能力有价值，但不能按 `Firefly` 原样搬进 `duet`。

## 6.1 评论系统

`Firefly` 式的评论能力可以借，但 `duet` 需要先定义自己的评论接口：

- 只允许在公开页面挂评论
- 私密页面不挂评论
- provider 通过配置接入，而不是把某个实现写死

因此评论系统应被视为“公共页面插件能力”，而不是博客模板默认组件。

## 6.2 公告、友链、相册、赞助等页面模块

这些页面模块适合进入 `duet`，但应以 `site` 域结构化内容形式接入，例如：

- `site/links`
- `site/announcement`
- `site/gallery`
- `site/support`

不能作为散落的主题页面逻辑存在。

换句话说，可吸收的是“页面类型”，不是“主题堆叠出来的页面实现”。

## 6.3 OG / 分享物料能力

`Firefly` 已经体现出更强的分享物料意识。`duet` 未来也值得做：

- 更稳定的 OG 图策略
- 分享卡片
- 二维码或海报式导出

但这些能力必须在 `public-only` 内容查询上构建，不能绕过 `duet` 的公开边界。

## 6.4 多语言支持

`Firefly` 的多语言支持值得借鉴，但 `duet` 当前不应立刻追求“全站翻译完成度”。

更合适的吸收方式是：

- 先保留配置与 schema 预留
- 后续按内容域逐步引入

即先吸收架构，不急着吸收完整实现复杂度。

## 7. 明确不建议吸收的能力

这些能力即使在主题里有效，也不适合作为 `duet` 主线目标。

## 7.1 重装饰型主题功能

例如：

- 音乐播放器
- 看板娘
- 樱花特效
- 广告位
- 大量漂浮装饰

这类功能会显著增加维护面，却不会增强 `duet` 作为内容引擎的核心能力。

## 7.2 回到博客中心型代码结构

`Firefly` 本质仍然是博客主题，因此很多能力默认都围绕文章详情页和博客首页展开。

`duet` 不能吸收这种结构性假设。后续的 `home / moments / blog / cv / portfolio` 必须继续保持内容域分离。

## 7.3 过度依赖主题配置开关

若所有能力都被做成主题开关，会带来两个问题：

1. 配置表面上更丰富，系统边界实际上更模糊
2. 后续 private-first 能力会被迫穿过大量主题逻辑

因此 `duet` 应只对“视觉与展示策略”开放配置，不对“系统契约”开放配置。

## 8. 建议路线图

## 8.1 P1：近期值得做

这些能力最符合当前阶段，且投入产出比最高。

1. 配置拆分
2. 布局原语整理
3. 阅读体验增强
4. 字体与主题配置整理

这一阶段的目标是：

- 降低单文件配置负担
- 提高站点完成度
- 为 `home` 和 `private posts` 奠定统一展示底层

## 8.2 P2：中期可做

1. 评论系统抽象
2. 公告 / 友链 / 相册等结构化页面模块
3. 更成熟的 OG / 分享物料策略

这一阶段的目标是：

- 让 `duet` 更像完整个人门户
- 但仍不引入后台、数据库与复杂应用心智

## 8.3 P3：谨慎评估后再做

1. 多语言内容深化
2. 作者后台配套展示层
3. 更复杂的展示特效与页面装饰

这一阶段必须建立在前两阶段稳定后进行，否则很容易让 `duet` 偏离主线。

## 9. 对当前 Duet 的具体建议

结合当前代码状态，建议先做以下几件事：

1. 把 `src/config.ts` 拆分为多模块配置目录
2. 抽象首页 / 归档 / moments 的列表布局原语
3. 保持 `src/app/*` 与 `src/domains/*` 的边界，不把主题功能塞回公共大文件
4. 所有未来新增页面模块都先落到 `site` 域结构化内容，而不是临时 hardcode
5. 所有搜索、订阅、索引能力继续只走公开查询

## 10. 完成标准

当以下条件成立时，可以认为 `duet` 已经成功吸收了 `Firefly` 中真正有价值的部分：

1. 配置系统已经按职责拆分，而不是继续堆在单一入口文件中
2. 页面布局能力已经抽象为可复用原语
3. 博客阅读体验得到增强，但没有回退到博客中心型架构
4. 新增站点模块采用结构化内容接入，而不是主题硬编码
5. private-first 边界在任何新增能力中都没有被削弱

## 11. 最终判断

`Firefly` 对 `duet` 的最大启发不是“功能很多”，而是：

- 一个主题如何被做成产品
- 一个模板如何形成稳定配置接口
- 一个个人站如何从“页面集合”变成“能力集合”

`duet` 应吸收这部分经验，但不能放弃自己的架构目标。

因此本设计的最终结论是：

- 借 `Firefly` 的工程化与产品化经验
- 不借 `Firefly` 的主题堆料路线
- 让 `duet` 继续朝 private-first 个人门户引擎演进
